# ДЕТАЛЬНЫЙ ПЛАН РЕАЛИЗАЦИИ: Self-Healing NLU MLOps

## ЭТАП 1: Фундамент, ETL и Безопасная очистка данных (Недели 1-2)
*Цель: Создать надежный конвейер извлечения и гарантировать 100% удаление PII до любых ML-операций.*

### Подзадача 1.1: Настройка Airflow и Extraction-пайплайна
- **Инструменты:** Apache Airflow, `httpx` (асинхронный HTTP-клиент), `asyncpg` (если источник PostgreSQL).
- **Реализация:** Создание кастомного Airflow Operator `ExtractFallbackLogsOperator`. Он должен поддерживать пагинацию API Just AI Caila или выполнять SQL-запрос с фильтром по `status IN ('fallback', 'transfer_to_operator')` и временному окну `execution_date`.
- **Действие:** Реализация механизма идемпотентности: использование `run_id` или хэша временного окна в качестве ключа, чтобы повторный запуск DAG не создавал дубликаты записей. Настройка ретраев (retry) при сетевых сбоях API.

### Подзадача 1.2: Модуль строгого маскирования PII
- **Инструменты:** `presidio-analyzer`, `presidio-anonymizer`, `spacy` (модель `ru_core_news_sm`).
- **Реализация:** Создание класса `PIIMaskingService`. Расширение стандартного `RecognizerRegistry` кастомными паттернами для РФ:
  - СНИЛС: `\b\d{3}[\s-]?\d{3}[\s-]?\d{3}\s?\d{2}\b`
  - ИНН физ. лица: `\b\d{12}\b`
  - Банковские карты: `\b\d{4}[\s-]?\d{4}[\s-]?\d{4}[\s-]?\d{4}\b`
  Настройка `OperatorConfig` для замены найденных сущностей на строгие токены (например, `[ТЕЛЕФОН_СКРЫТ]`, `[ПАСПОРТ_СКРЫТ]`).
- **Действие:** Написание модульных тестов (pytest) с набором из 50+ граничных случаев (опечатки в номерах, разные форматы дат), гарантирующих >95% точность маскирования перед передачей данных дальше.

### Подзадача 1.3: Эвристическая очистка "мусора" (Data Cleaning)
- **Инструменты:** `pymorphy3` или `natasha`, регулярные выражения, `pandas`.
- **Реализация:** Построение пайплайна фильтрации:
  1. Отбрасывание фраз короче 2 значимых слов.
  2. Регулярная фильтрация явных артефактов ASR (например, строки, состоящие более чем на 50% из повторяющихся символов или не-кириллицы).
  3. Лемматизация оставшихся фраз для улучшения качества последующей векторизации.
- **Действие:** Реализация функции `clean_and_validate_text()`, которая возвращает `Optional[str]`. Если текст не проходит валидацию, он логируется с причиной (`drop_reason`) и исключается из датасета для кластеризации.

## ЭТАП 2: Векторизация и Кластеризация (Недели 3-4)
*Цель: Преодолеть "ловушку качества данных" и получить стабильные, осмысленные кластеры из зашумленных текстов.*

### Подзадача 2.1: Асинхронная векторизация (Embeddings)
- **Инструменты:** `sentence-transformers`, модель `ai-forever/ru-en-RoSBERTa` (или `multilingual-e5-large`), `torch`.
- **Реализация:** Создание сервиса `EmbeddingService`. Критически важна обработка батчами (например, `batch_size=64` или `128` в зависимости от доступной VRAM) для предотвращения Out-Of-Memory (OOM) ошибок. Использование `torch.no_grad()` для экономии памяти.
- **Действие:** Реализация асинхронного или многопоточного (`concurrent.futures`) вызова модели внутри Airflow Task, с сохранением полученных векторов во временное хранилище (например, Parquet-файл в S3 или оперативную память, если объем позволяет).

### Подзадача 2.2: Настройка и выполнение HDBSCAN
- **Инструменты:** `hdbscan`, `numpy`, `pandas`.
- **Реализация:** Инициализация модели `hdbscan.HDBSCAN`. Ключевая калибровка гиперпараметров для баланса между детализацией и шумом:
  - `min_cluster_size`: 15 (минимальное количество фраз для признания интента реальным).
  - `min_samples`: 5 (для определения границ кластера и отсечения выбросов).
  - `metric`: 'cosine' (так как используем косинусное сходство эмбеддингов).
- **Действие:** Написание скрипта кластеризации, который присваивает каждой фразе метку кластера (целое число) или `-1` (шум/outlier).

### Подзадача 2.3: Пост-фильтрация и валидация кластеров
- **Инструменты:** `sklearn.metrics` (Silhouette score), кастомная эвристика.
- **Реализация:** Разработка класса `ClusterValidator`. Он отбраковывает:
  1. Кластеры с размером < `min_cluster_size`.
  2. Кластеры, помеченные как шум (`-1`).
  3. Кластеры с низким внутренним сходством (Silhouette score < 0.2).
  4. Кластеры, состоящие исключительно из фраз-паразитов ("алло", "да", "спасибо", "я не понял").
- **Действие:** Формирование финального списка объектов `ValidCluster`, содержащего `cluster_id`, `centroid_embedding`, и список из топ-10 наиболее репрезентативных фраз (closest to centroid) для передачи в LLM.

## ЭТАП 3: LLM-агент и Генерация данных (Недели 5-6)
*Цель: Превратить сырые кластеры в структурированные обучающие данные с помощью LLM.*

### Подзадача 3.1: Проектирование промпта и строгой схемы вывода
- **Инструменты:** YandexGPT Pro API (или локальный vLLM с Llama-3), `pydantic` V2.
- **Реализация:** Определение строгой Pydantic-модели:
  ```python
  class GeneratedIntent(BaseModel):
      intent_name: str = Field(description="Краткое название интента на русском, snake_case")
      description: str = Field(description="Описание намерения клиента в 1 предложении")
      utterances: list[str] = Field(min_items=20, max_items=30, description="Вариации фраз")
  ```
  Системный промпт должен требовать возврата **строго валидного JSON** без markdown-оберток, предоставляя топ-10 примеров из кластера как контекст.
- **Действие:** Создание шаблона промпта и функции-парсера, которая пытается десериализовать ответ LLM через `GeneratedIntent.model_validate_json()`.

### Подзадача 3.2: Механизм валидации и Retry (Self-Correction)
- **Инструменты:** `tenacity` (для retry), `pydantic` ValidationError.
- **Реализация:** Обертка вызова LLM. Если `model_validate_json` выбрасывает ошибку (например, LLM сгенерировал 5 фраз вместо 20, или невалидный JSON), система формирует уточняющий промпт: *"Ты допустил ошибку: {error_message}. Исправь JSON и верни только его, убедившись, что фраз не менее 20"*.
- **Действие:** Настройка декоратора `@retry(stop=stop_after_attempt(2), wait=wait_fixed(1))` исключительно на исключения валидации Pydantic.

### Подзадача 3.3: Асинхронная обработка и контроль нагрузки
- **Инструменты:** `asyncio`, `httpx`, `asyncio.Semaphore`.
- **Реализация:** Поскольку кластеров может быть много, одновременный вызов LLM для всех приведет к Rate Limiting. Внедрение семафора (например, `max_concurrent_requests=5`) для контроля параллелизма. Настройка `temperature=0.7` для баланса между разнообразием генерируемых фраз и сохранением исходного смысла кластера.
- **Действие:** Реализация асинхронной функции `process_clusters_batch()`, которая безопасно и эффективно обрабатывает все валидные кластеры за один проход DAG.

## ЭТАП 4: HITL, Интеграции и Экспорт (Недели 7-8)
*Цель: Обеспечить удобный и безопасный процесс утверждения изменений человеком.*

### Подзадача 4.1: Генератор экспортных файлов
- **Инструменты:** `pandas`, `json`, `boto3` / `aiobotocore` (для S3).
- **Реализация:** Преобразование списка одобренных `GeneratedIntent` в формат, нативно поддерживаемый Just AI Caila (обычно CSV или специфичный JSON с полями `intent`, `utterance`, `weight`).
- **Действие:** Реализация функции, которая сохраняет файл в S3, генерирует Presigned URL (с TTL 7 дней) и возвращает ссылку для использования в задаче Битрикс24.

### Подзадача 4.2: Интеграция с Битрикс24 (Создание задачи на апрув)
- **Инструменты:** `httpx`, Bitrix24 REST API (`task.item.add`), `pybreaker` (Circuit Breaker).
- **Реализация:** Формирование богатого HTML-описания задачи:
  - Заголовок: `[AI] Новый интент: {intent_name}`
  - Тело: Описание, топ-3 реальных примера из логов, топ-3 сгенерированных примера, кнопка/ссылка на скачивание полного датасета из S3.
- **Действие:** Написание асинхронного клиента `Bitrix24TaskService` с обработкой специфичных ошибок Битрикса (ошибка внутри HTTP 200) и механизмом повторных попыток.

### Подзадача 4.3: Уведомление аналитика (Telegram)
- **Инструменты:** Telegram Bot API (`httpx`).
- **Реализация:** Отправка краткого, информативного сообщения в рабочий чат или лично аналитику сразу после успешного создания задачи в Битрикс24.
- **Действие:** Реализация `TelegramNotifier`, который отправляет сообщение вида: *"🤖 Найдено {N} новых потенциальных интентов. Задача на апрув создана: {ссылка_на_битрикс}"*.

## ЭТАП 5: Production Hardening и IaC (Неделя 9)
*Цель: Обеспечить надежность, масштабируемость и наблюдаемость системы.*

### Подзадача 5.1: Контейнеризация и Infrastructure as Code
- **Инструменты:** Docker (multi-stage), Terraform, Helm.
- **Реализация:** 
  - Создание `Dockerfile` для Airflow Worker, который включает установку `ru_core_news_sm` и системных зависимостей на этапе сборки, чтобы избежать загрузок во время выполнения DAG.
  - Написание Terraform-модулей для развертывания Managed PostgreSQL, S3-бакета и K8s-кластера с установленным Airflow (через официальный Helm chart).
- **Действие:** Настройка CI/CD пайплайна (GitHub Actions/GitLab CI) для автоматической сборки образа, сканирования уязвимостей (Trivy) и деплоя в K8s.

### Подзадача 5.2: Сквозная наблюдаемость (OpenTelemetry)
- **Инструменты:** `opentelemetry-api`, `opentelemetry-sdk`, `opentelemetry-instrumentation-httpx`.
- **Реализация:** Инструментация кастомных Airflow Operators. Генерация `trace_id` на старте DAG и передача его в заголовки всех внешних вызовов (Caila API, LLM API, Bitrix24). Экспорт трейсов в Grafana Tempo или Jaeger.
- **Действие:** Настройка `webserver_config.py` и переменных окружения Airflow для включения OTel. Создание дашборда в Grafana, отображающего время выполнения каждого шага (Extract -> Clean -> Embed -> Cluster -> LLM).

### Подзадача 5.3: Алертинг и мониторинг деградации
- **Инструменты:** Prometheus, Alertmanager.
- **Реализация:** Определение правил (Prometheus Rules) для критических сценариев:
  1. `AirflowDagFailed`: DAG не выполнился за отведенное время.
  2. `PIILeakDetected`: (Кастомная метрика) если эвристическая проверка на выходе LLM обнаружила паттерн, похожий на телефон или карту.
  3. `ZeroValidClusters`: HDBSCAN вернул 0 кластеров 3 дня подряд (сигнал о резком изменении формата входящих логов или поломке экстрактора).
- **Действие:** Настройка `prometheus-rules.yaml` и маршрутизация алертов из Alertmanager в Telegram-чат команды MLOps.
